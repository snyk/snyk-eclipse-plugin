1. **[high]** `EclipseThemeCssProvider.java`: Four CSS variables (`focusBorder`, `listHoverBg`, `listSelectionBg`, `buttonBg`) all map to `SWT.COLOR_LIST_SELECTION`. This is semantically incorrect, not merely redundant. Focus borders and button backgrounds are visually distinct elements — mapping a button background to the list-selection color will produce a visually broken result in themes where those colors differ (e.g., dark themes with a blue selection and a grey widget background). This isn't simplification; it's the wrong color applied four times.

2. **[medium]** `EclipseThemeCssProvider.java`: `resolveFontFamily` and `resolveFontSize` each independently execute `display.getSystemFont().getFontData()[0]`, including independent null guards and try/catch blocks. Both are called from `buildStyleBlock` in sequence. `getFontData()` returns an array; calling it twice to extract two fields of `getFontData()[0]` is redundant work wrapped in doubled boilerplate. A single `resolveFontData` method returning `FontData` (or null) collapses both methods to one lookup and eliminates the duplicated defensive structure.

3. **[medium]** `TreeViewBrowserHandler.java`: The three-branch injection (`${ideStyle}` → `</head>` → prepend) exists to handle HTML that may not contain `${ideStyle}`. The stated key fact is that the LS template *always* has `${ideStyle}`, confirmed by an LS-side test assertion. The two fallback branches are unreachable in production and generate 5 test methods covering scenarios that cannot occur. Each test method for the fallback paths costs maintenance against a guarantee held by upstream. Delete the fallback branches; make the injection a single string replace and a log/exception if the token is absent.

4. **[low]** `TreeViewBrowserHandler.java`: `injectThemeCss` is `public static` with exactly one caller: `setBrowserText` in the same class. `buildStyleBlock` is `public static` with one caller: `injectThemeCss`. The tests access both directly because they're public, but package-private would suffice for tests in the same package. Public static signals reusable utility API — neither method is that.

5. **[low]** `EclipseThemeCssProviderTest` (implied): Four `buildStyleBlock(null)` tests verify CSS variable *names and structure* with a null display, meaning color resolution falls back to defaults and font resolution falls back to hardcoded values. These tests confirm the string template is well-formed but do not verify that any Eclipse theme color is actually injected correctly. They provide structural confidence while leaving the actual outcome — correct color values from a real display — untested.

6. **[low]** `EclipseThemeCssProviderTest`: Four test methods for `toHex` (black, white, red, arbitrary) exhaustively cover a hex-formatting one-liner. A single parameterized test or two boundary checks (black = `#000000`, white = `#ffffff`) would provide the same signal. The current coverage is disproportionate to the complexity and fragility risk of the method.
