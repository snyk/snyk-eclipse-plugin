Relevant principles:

### boundary-discipline
Place validation, type narrowing, and error handling at system boundaries. Trust internal code unconditionally.
- Validate at the point of entry — exactly once
- Business logic: pure functions, no framework dependencies
- Application shell (routing, event handling, external integrations): thin and mechanical
- If data is crossing a system boundary, handle it there. If logic can be a pure function independent of framework concerns, it should be.

### foundational-thinking
Structural decisions (data models, phase ordering, infrastructure) optimize for option value. Code-level decisions (helpers, abstractions, patterns) optimize for simplicity.
- Bad complexity: helper classes for one-time operations, indirection layers.
- Good complexity: choosing the data structure that matches actual access patterns.
- DRY at the structural level (types, data models) — but three similar lines of code is better than a premature abstraction.
- Explicit over clever.
- Concurrency corollary: Before sharing state between actors, ask: "What happens if another actor modifies this concurrently?" If the answer isn't "nothing", isolate.
