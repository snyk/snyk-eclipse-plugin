1. **[HIGH] Double invocation of `treeBrowserHandler.selectNode` per `selectTreeNode(Issue,String)` call.**
   `SnykToolView.java:538` posts `asyncExec(() -> treeBrowserHandler.selectNode(issue.id()))` explicitly. Concurrently, `updateSelection` at line 558 posts a second `asyncExec` that calls `treeViewer.setSelection`, which fires `selectionChanged` synchronously on the UI thread. The `selectionChanged` listener (line 138) wraps its body in yet another `asyncExec`, and that body calls `treeBrowserHandler.selectNode(issueTreeNode.getIssue().id())` at line 149. Both asyncExecs land in the UI event queue and execute, producing two JS calls to `__selectTreeNode__` per invocation — a structural defect caused by the method conflating JFace tree selection (which has selection-listener side effects) with HTML tree selection.

2. **[HIGH] Data race on non-volatile `treeBrowserHandler` field read from background threads.**
   `treeBrowserHandler` is declared at `SnykToolView.java:72` without `volatile`. It is written once on the UI thread at line 110 (`createPartControl`). It is read without synchronization from background threads in `selectTreeNode(Issue,String)` (line 537) and `updateTreeViewHtml` (line 566). The JVM memory model provides no happens-before guarantee between the UI-thread write and these bg-thread reads. The field must be `volatile` or reads must be confined to the UI thread (or marshalled via `syncExec`).

3. **[HIGH] Incomplete JS escaping in `TreeViewBrowserHandler.selectNode` — newlines and control characters unhandled.**
   `TreeViewBrowserHandler.java:37` escapes only `\` and `'`. A carriage return or newline in `issueId` (e.g. `CR`, `LF`) terminates the single-quoted JS string literal, yielding a syntax error rather than a safe no-op. Language server–sourced issue IDs must be treated as untrusted at this boundary. The fix is to escape all non-printable characters (at minimum `\n`, `\r`, `\u2028`, `\u2029`) or use `JSON.stringify` interop rather than manual string concatenation.

4. **[MEDIUM] `pendingHtml` defensive path is dead code that introduces false threading complexity.**
   `openToolView` always dispatches via `syncExec`, which guarantees `createPartControl` completes — and thus `treeBrowserHandler` is non-null — before any caller can invoke `updateTreeViewHtml`. The null-guard at `SnykToolView.java:566-568`, the flush at lines 113-117, and the `volatile` annotation at line 74 are therefore unreachable. The `volatile` keyword in particular implies a threading contract to future readers that does not exist and may invite incorrect reasoning about memory visibility elsewhere in the class.

5. **[MEDIUM] `selectTreeNode(Issue,String)` violates single-responsibility and makes the double-fire invisible from the interface boundary.**
   `SnykToolView.java:534-540` conflates three distinct operations: resolving the JFace model node, setting JFace selection (which has listener side effects), and issuing a JS call to the HTML browser. Because the listener at line 133 independently mirrors JFace selection to the HTML browser, the method's explicit JS call at line 538 is redundant and harmful. This tight coupling between JFace selection mechanics and HTML-browser selection is invisible to callers via `ISnykToolView`, making the double-fire impossible to diagnose without understanding both sides of the listener registration.

6. **[LOW] `selectTreeNode(String)` added to `ISnykToolView` with no external callers.**
   `ISnykToolView.java:143` declares `void selectTreeNode(String issueId)`. No call site outside `SnykToolView` uses this method. Adding interface methods for anticipated future use expands the contract that all mocks and test doubles must implement, and signals a boundary concern (the interface now mixes legacy-tree operations with HTML-browser operations) without any present justification.

7. **[LOW] `ISnykToolView` continues to leak `TreeViewer` across the interface boundary.**
